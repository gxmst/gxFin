# 轻量量化模拟交易与回测系统

> 技术设计文档（最终版）

## 1. 设计目标

本文档定义一个面向**单用户**的轻量量化模拟系统技术方案。

系统要求：

- 在线读取市场数据
- 支持单策略回测
- 支持单策略低频模拟执行
- 可部署到 VPS 并远程访问
- UI 简洁、好看、可用
- 整体资源占用保持克制

本项目不是高频交易系统，也不是多用户交易平台。

## 2. 总体架构原则

### 2.1 先保证语义正确

系统最重要的不是框架复杂度，而是以下语义必须正确：

- 使用已闭合 K 线生成信号
- 使用下一根 K 线开盘价成交
- 手续费与滑点计算正确
- 持仓、现金、权益更新正确
- 同一根 K 线不重复处理

### 2.2 先轻量可用，再逐步增强

第一版优先追求：

- 代码结构清晰
- 功能闭环完整
- 易部署
- 易维护
- 资源占用可控

不为未来可能出现的大型场景提前过度设计。

### 2.3 单用户、单策略、低频优先

默认运行形态为：

- 单用户
- 单策略
- 单实例
- 少量标的
- 低频周期

系统主线应以“策略执行”作为核心，以“回测验证”作为辅助，不应把第一版实现成复杂回测平台。

### 2.4 UI 与执行器逻辑分离

UI 负责配置、展示、触发控制；执行器负责取数、计算、模拟成交、落库。

可以放在同一仓库，但不建议把执行主循环直接写进 UI 页面逻辑。

## 3. 推荐系统结构

### 3.1 模块划分

建议保留以下 6 个核心模块：

1. `ui`：页面展示与用户控制
2. `runtime`：运行调度与任务入口
3. `strategy`：策略接口与策略实现
4. `broker`：模拟成交与账户计算
5. `data`：行情获取、标准化、缓存
6. `storage`：SQLite 持久化

这个结构已经足够支持第一版，不需要再做更重的分层。

### 3.2 推荐目录

```text
project/
  app/
    ui/
    runtime/
    strategy/
    broker/
    data/
    storage/
    config/
  tests/
  docker/
```

如果需要再细分，也应保持克制。

## 4. 数据层设计

### 4.1 数据源

推荐：

- CCXT：作为加密数据主来源
- yfinance：作为股票 / ETF 可选来源

第一版应优先把加密市场跑通。

### 4.2 标准数据接口

所有数据源统一输出标准 K 线结构：

- `timestamp`
- `open`
- `high`
- `low`
- `close`
- `volume`
- `symbol`
- `timeframe`
- `source`

建议接口形式：

```python
class BaseDataProvider:
    def fetch_ohlcv(
        self,
        symbol: str,
        timeframe: str,
        since: int | None = None,
        limit: int | None = None,
    ) -> pd.DataFrame:
        ...
```

### 4.3 数据处理原则

- 内部统一使用 UTC 时间
- 只对已闭合 K 线生成有效信号
- 优先增量拉取，而不是频繁全量拉取
- 拉取后做去重、排序、基础校验

### 4.4 缓存策略

缓存只做轻量方案：

- 每个标的 / 周期只缓存最近一段数据
- 缓存文件或缓存表都可以
- 支持上限控制
- 支持自动清理旧数据

不建议第一版做大而全的历史仓库。

## 5. 策略层设计

### 5.1 策略接口

建议统一策略基类：

```python
class BaseStrategy:
    name: str
    description: str
    default_config: dict

    def __init__(self, config: dict):
        self.config = config

    def validate_config(self) -> None:
        ...

    def generate_signal(self, df: pd.DataFrame) -> str:
        ...
```

推荐策略组织方式：

- 每个策略放在独立 Python 文件中
- 策略文件存放在固定目录，例如 `strategy/strategies/`
- 系统启动时发现或注册可用策略
- UI 或配置中选择当前启用的单个策略
- 第一版不做多策略同时执行

### 5.2 第一版策略范围

第一版只实现少量简单策略即可，例如：

- 双均线策略
- RSI 策略

策略输出保持简单：

- `BUY`
- `SELL`
- `HOLD`

不建议第一版输出复杂订单意图对象。

## 6. 模拟执行与账户设计

### 6.1 成交语义

这是系统的核心规则：

1. 使用当前已闭合 K 线生成信号
2. 若产生交易动作，则在下一根 K 线开盘价成交
3. 成交价应用滑点
4. 成交时扣除手续费
5. 更新账户与持仓
6. 写入交易记录和权益快照

### 6.2 第一版执行模式

保持简单：

- `BUY`：全仓买入
- `SELL`：全部卖出
- `HOLD`：不操作

不实现：

- 部分成交
- 限价单
- 订单撤销
- 分批建仓
- 多仓位层级管理

### 6.3 账户模型

账户至少维护：

- `cash`
- `position_qty`
- `position_avg_price`
- `position_value`
- `total_equity`
- `realized_pnl`
- `unrealized_pnl`

### 6.4 防重复处理

必须防止同一根 K 线重复触发成交。

建议至少记录：

- 最近处理过的 bar 时间
- 最近一次信号时间
- 最近一次成交时间

可以存数据库，也可以存轻量运行状态表。

## 7. 回测设计

### 7.1 回测原则

回测必须复用与模拟执行一致的成交语义，不允许回测和实时执行各用一套规则。

回测在本项目中的定位是辅助验证策略，而不是第一版的核心产品形态。

### 7.2 回测输入

- 标的
- 周期
- 时间范围
- 初始资金
- 手续费
- 滑点
- 策略参数

### 7.3 回测输出

- 总收益率
- 最大回撤
- 交易次数
- 胜率
- 资金曲线
- 交易记录

第一版不做复杂参数优化器，不做大规模并行回测。

## 8. 存储设计

### 8.1 存储选择

第一版主存储使用 SQLite。

原因：

- 轻量
- 易部署
- 单用户场景足够
- 很适合 VPS 小型系统

### 8.2 建议保存的数据

只保留必要内容：

- `account_equity`
- `trade_history`
- `backtest_results`
- `system_logs`
- `runtime_state`

如果实现需要，也可增加轻量缓存表。

### 8.3 数据库原则

- 表数量尽量少
- 字段保持实用
- 避免过度抽象
- 写入逻辑尽量简单
- UI 查询只查必要数据

## 9. UI 设计

### 9.1 UI 目标

UI 应做到：

- 简洁
- 好看
- 易用
- 远程访问友好

但不追求复杂平台型控制台。

### 9.2 页面内容

第一版建议只保留一个主页面，或者一个主页面加一个回测结果页。

主页面建议包含：

- 标的选择
- 周期选择
- 当前策略选择
- 参数编辑
- 启动 / 停止按钮
- 回测按钮
- 当前账户指标卡片
- K 线图
- 资金曲线图
- 最近交易记录
- 最近日志

### 9.3 UI 刷新策略

UI 可以常驻，但刷新应克制。

建议：

- 固定低频刷新，例如 10 到 30 秒
- 不做高频自动刷新
- 不在每次页面 rerun 时重新拉全量数据
- 图表只展示必要时间范围
- 日志只展示最近少量记录

### 9.4 UI 技术取舍

推荐使用：

- Streamlit
- Plotly

原因：

- Python 一体化开发效率高
- 足够做漂亮的个人控制台
- 对金融图表友好

但实现时要避免：

- 页面挂太多图
- 大量表格同时加载
- 页面逻辑直接承担执行主循环
- 为了“功能齐全”而堆太多次要面板

## 10. 运行与调度设计

### 10.1 运行模式

系统支持两种模式：

- 手动回测
- 低频模拟运行

### 10.2 模拟运行方式

第一版调度不需要太复杂，但应稳定可用。

建议：

- 用简单明确的周期调度
- 每次触发时再次校验 bar 是否闭合
- 不依赖 UI 页面触发执行主逻辑

如果使用 APScheduler，可以只用最基本能力，不要做复杂任务编排。

### 10.3 健康检查

只保留轻量版：

- 当前状态
- 最近更新时间
- 最近执行时间
- 最近错误

不做完整监控平台。

## 11. 部署设计

### 11.1 部署目标

第一版需要同时支持：

- 本地运行
- VPS 运行
- Docker 部署

### 11.2 Docker 的作用

容器化主要用于：

- 统一环境
- 简化部署
- 便于重启
- 便于保活

不是为了做复杂集群编排。

### 11.3 远程访问

支持浏览器远程访问即可。

安全控制先保持轻量，例如：

- 基础密码保护
- 反向代理
- 基础访问限制

## 12. 性能与资源控制

### 12.1 总体判断

本项目应被设计成一个低负载系统，而不是高性能系统。

### 12.2 资源控制原则

- 少量标的
- 单策略
- 低频执行
- 小缓存
- 低频 UI 刷新
- 少图表
- 少表格
- 少后台进程

### 12.3 实现注意事项

实现时应特别避免：

- 反复复制大 DataFrame
- 无上限缓存历史数据
- UI 每次刷新都重新做重计算
- 日志长期堆在内存中
- 大量无必要的后台组件

## 13. 最终结论

最终推荐的实现方向是：

- 用 Python 做统一主语言
- 用 CCXT 作为主数据源
- 用 SQLite 做轻量持久化
- 用 Streamlit + Plotly 做简洁美观的 UI
- 用单策略、低频轮询、本地模拟成交构成最小闭环
- 用 Docker 支持 VPS 部署和保活

整个系统应优先做成一个**单用户、在线取数、轻量可远程使用、资源克制但体验不错**的个人量化研究工具。
