# 轻量量化模拟交易与回测系统

> 产品需求文档（最终版）

## 1. 项目定位

### 1.1 项目目标

构建一个面向**单用户**的轻量量化研究与模拟交易系统。

系统支持在线获取行情数据，在本地或 VPS 上运行单策略模拟执行与历史回测，并通过一个简洁、好看的 Web UI 展示核心状态与结果。

本项目的目标不是做专业交易平台，也不是做高频系统，而是做一个：

- 个人可长期使用的策略实验工具
- 可在线取数的轻量虚拟盘执行器
- 可远程访问的个人研究控制台
- 可部署到 VPS 并通过容器稳定运行的小型系统

### 1.2 明确边界

本系统当前明确不做以下事情：

- 不接入真实券商或交易所下单
- 不支持多用户协作
- 不做多策略并发调度
- 不做高频交易
- 不做复杂订单簿、逐笔撮合、部分成交模型
- 不做企业级权限、审计、风控体系

所有成交、资金、持仓、手续费、滑点、收益均在本地模拟计算。

### 1.3 使用方式

系统的典型使用方式如下：

- 用户在本地编写或维护策略文件
- 用户在本地或远程浏览器打开 UI
- 选择交易标的、时间周期和当前启用策略
- 系统在线拉取行情数据
- 用户可手动触发回测验证策略，也可启动低频模拟运行
- 系统基于策略信号进行本地模拟成交
- UI 展示核心账户状态、K 线信号与收益结果

## 2. 总体设计原则

### 2.1 功能做减法

系统以“够用、稳定、简单”为优先，不追求一次做成大而全平台。

### 2.2 保证执行语义正确

重点保证：

- 数据时间顺序正确
- K 线闭合判断正确
- 信号生成与成交时点正确
- 手续费、滑点、持仓和收益计算正确

### 2.3 单用户优先

所有设计默认围绕单用户场景展开，不为多人协作和复杂并发做额外设计。

### 2.4 资源占用可控

系统应控制内存和 CPU 占用，避免引入不必要的后台组件和大规模缓存。

### 2.5 UI 简洁但美观

UI 不追求“大屏监控”式复杂布局，而是追求：

- 页面清晰
- 信息密度适中
- 图表少而重要
- 视觉风格统一
- 可远程查看

## 3. 核心用户场景

### 3.1 策略执行场景

用户在本地维护策略文件，并在系统中选择其中一个作为当前启用策略。系统部署到 VPS 后，按设定周期在线取数并执行该策略，通过虚拟账户模拟买卖并记录结果。

### 3.2 回测场景

用户选择一个标的、一个时间周期、一组策略参数，系统拉取对应历史数据并执行回测，输出收益、回撤、交易记录和图表结果。

### 3.3 轻量模拟运行场景

用户选择标的与策略后，系统按低频周期在线取数并模拟执行，例如 `5m`、`15m`、`1h`、`4h`、`1d`。

### 3.4 远程查看场景

系统部署在 VPS 上后，用户可通过浏览器远程查看当前运行状态、最近信号、资金变化和核心图表。

## 4. MVP 范围

第一版只实现最重要、最能用的一组能力。

### 4.1 必做功能

- 在线行情数据获取
- 历史 K 线缓存
- 单策略回测
- 单策略模拟执行
- 本地模拟账户
- 手续费与滑点处理
- 核心交易记录
- 核心账户状态展示
- 简洁 Web UI
- 远程访问支持
- Docker 部署支持

### 4.2 可做但保持轻量

- 简单健康检查
- 基础日志
- 缓存上限与自动清理
- 简单运行状态页
- 简单导出能力

### 4.3 第一版不做

- 多账户
- 多策略并行
- 复杂消息通知
- 大规模参数优化
- 复杂权限系统
- 复杂订单生命周期管理
- WebSocket 实时流

## 5. 功能需求

### 5.1 行情数据

系统应支持从在线数据源读取行情，并统一为标准 K 线结构。

要求如下：

- 支持加密市场数据
- 可选支持股票 / ETF 数据
- 支持按标的、周期、时间范围拉取历史 K 线
- 支持按运行周期增量刷新最新数据
- 支持本地缓存，减少重复请求
- 支持缓存大小上限控制
- 支持清理过旧缓存

建议支持的周期：

- `5m`
- `15m`
- `1h`
- `4h`
- `1d`

默认不优先做 `1m` 高频场景。

### 5.2 策略系统

第一版只需要支持**单策略运行**，但代码结构应保留后续扩展空间。

策略的使用方式应尽量贴近个人开发习惯：

- 每个策略可以是一个独立的 Python 文件
- 策略文件放在固定目录中
- 系统启动后可发现或加载这些策略
- 用户可在 UI 中选择当前启用的策略
- 第一版同一时刻只执行一个策略

策略模块至少需要支持：

- 策略基类
- 策略名称和描述
- 默认参数
- 参数校验
- 信号输出：`BUY` / `SELL` / `HOLD`

第一版建议优先支持 1 到 2 个简单策略模板，例如：

- 双均线策略
- RSI 策略

### 5.3 模拟执行

系统必须支持本地模拟成交。

核心规则如下：

- 信号基于已闭合 K 线生成
- 成交不能使用信号当根 K 线收盘价
- 成交使用下一根 K 线开盘价作为参考价
- 统一加入滑点
- 统一扣除手续费

第一版执行模式保持简单：

- `BUY`：全仓买入
- `SELL`：全部卖出
- `HOLD`：不操作

### 5.4 模拟账户

系统应维护一个轻量虚拟账户，至少包括：

- 初始资金
- 当前现金
- 当前持仓数量
- 当前持仓成本
- 当前总资产
- 已实现盈亏
- 未实现盈亏

### 5.5 回测

系统应支持单策略历史回测。

回测的定位是：用于辅助验证策略，而不是把第一版做成一个复杂回测平台。

回测输入至少包括：

- 标的
- 周期
- 起始时间
- 结束时间
- 初始资金
- 手续费
- 滑点
- 策略参数

回测输出至少包括：

- 总收益率
- 最大回撤
- 交易次数
- 胜率
- 资金曲线
- 交易明细

### 5.6 UI

UI 是系统的重要部分，但第一版页面必须克制。

建议保留以下核心区域：

- 顶部状态栏
- 左侧控制区
- 右侧主展示区

UI 至少应支持：

- 选择标的
- 选择周期
- 选择当前策略
- 编辑策略参数
- 触发回测
- 启动 / 停止模拟运行
- 展示账户核心指标
- 展示 1 个 K 线图
- 展示 1 个资金曲线图
- 展示最近交易记录
- 展示最近运行日志

UI 设计要求：

- 页面风格统一
- 图表数量控制在最重要的 1 到 2 个
- 刷新频率不要过高
- 不一次性加载过多历史数据
- 不做复杂大屏监控布局
- 不为了展示而堆叠过多模块

### 5.7 远程访问

系统应支持部署到 VPS，并通过浏览器远程访问。

第一版只需提供轻量保护能力，例如：

- 简单密码保护
- 反向代理后访问
- 基础访问隔离

### 5.8 健康检查

健康检查只做基础版即可，至少包含：

- 最近一次数据更新时间
- 最近一次执行时间
- 当前运行状态
- 最近一次错误信息

不做复杂监控和告警系统。

### 5.9 存储

第一版存储应尽量简单，以轻量可维护为主。

建议存储内容包括：

- 交易记录
- 账户权益快照
- 回测结果摘要
- 关键运行日志
- 必要的缓存索引

第一版不要求设计过多复杂数据表。

### 5.10 缓存与清理

系统应支持简单缓存策略，避免无上限增长。

至少包括：

- 每个标的 / 周期缓存最近一定范围的数据
- 支持缓存容量上限
- 支持按时间或数量清理过期缓存
- UI 不默认加载全量历史缓存

## 6. 技术选型要求

### 6.1 推荐技术栈

- 语言：Python
- UI：Streamlit
- 图表：Plotly
- 加密数据：CCXT
- 股票数据：yfinance（可选）
- 数据处理：Pandas、NumPy
- 存储：SQLite
- 部署：Docker

### 6.2 技术约束

实现时应遵循以下原则：

- 不引入 Redis、Celery、PostgreSQL 等重型组件
- 不做复杂微服务拆分
- 不做高频实时推送
- 不将大量运行状态长期保存在内存中
- 尽量减少大 DataFrame 的重复复制

## 7. 性能与资源预期

本项目不是高负载系统。

### 7.1 预期负载

典型运行方式：

- 单用户
- 单策略
- 少量标的
- 低频周期
- 低频 UI 刷新

### 7.2 资源目标

在设计合理、实现克制的前提下：

- 纯执行器应尽量控制在较低内存占用
- 带 UI 的完整运行形态应保持轻量
- CPU 大部分时间应处于空闲或低占用状态

目标不是极限压榨到固定数字，而是保证系统在普通轻量 VPS 上长期稳定运行。

## 8. 部署要求

### 8.1 运行环境

支持以下环境：

- 本地开发环境
- Linux VPS 运行环境

### 8.2 部署方式

第一版应支持：

- 本地直接运行
- Docker 容器运行

### 8.3 容器目标

容器化主要用于：

- 简化部署
- 环境一致
- 便于保活和重启

## 9. 最终交付目标

最终希望得到的是一个：

- 单用户可用
- 在线取数
- 回测和模拟执行都可用
- 远程可访问
- UI 简洁好看
- 资源消耗较低
- 易于继续迭代

的个人量化实验系统。
